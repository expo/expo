In this chapter, we'll create our initial workflow to use `preview` build profiles with fingerprints, add Slack notifications, and set up a separate workflow for pull request (PR) previews using OTA.

## Using `build` job type for preview builds

Preview builds are used for team and stakeholder testing. Teams distribute them via internal distribution so members can test the changes on real devices or emulators/simulators before a production release. They don't have to run a development server on their local machine or check a Git branch to test the changes.

<PreviewPipeline />

Like development builds, we will add fingerprints to avoid unnecessary rebuilds when native code doesn't change.

<Step label="1">

### Add a preview.yml

Inside **.eas/workflows/**, let's add a new file called **preview.yml**. This workflow file will use `preview` build profile.

```yaml preview.yml
name: Preview builds

jobs:
  fingerprint:
    name: Fingerprint
    type: fingerprint
    # @info #
    environment: preview
    # @end #
  get_android_build:
    name: Check for existing Android build
    needs: [fingerprint]
    type: get-build
    params:
      fingerprint_hash: ${{ needs.fingerprint.outputs.android_fingerprint_hash }}
      # @info #
      profile: preview
      # @end #
  get_ios_build:
    name: Check for existing iOS build
    needs: [fingerprint]
    type: get-build
    params:
      fingerprint_hash: ${{ needs.fingerprint.outputs.ios_fingerprint_hash }}
      # @info #
      profile: preview
      # @end #
  build_android:
    name: Build Android
    needs: [get_android_build]
    if: ${{ !needs.get_android_build.outputs.build_id }}
    type: build
    params:
      platform: android
      # @info #
      profile: preview
      # @end #
  build_ios:
    name: Build iOS
    needs: [get_ios_build]
    if: ${{ !needs.get_ios_build.outputs.build_id }}
    type: build
    params:
      platform: ios
      # @info #
      profile: preview
      # @end #
```

</Step>

<Step label="2">

### Run the workflow

Run the workflow manually using the following command:

<Terminal cmd={['$ eas workflow:run .eas/workflows/preview.yml']} />

In EAS dashboard, notice that the workflow follows the same fingerprint pattern we implemented for development builds. Since we are creating the preview build for the first time, it runs the build jobs for Android and iOS. If we push another commit to `main` branch without changing native code, the workflow will skip the build jobs since compatible builds already exist.

## Slack notifications

> **info** This section is optional. If you don't use Slack or want to skip this part, you can jump to the next section about PR previews.

Slack notifications are popular with teams that are distributed. When a build completes (or skips), the team can be notified within the Slack by using [`slack`](/eas/workflows/pre-packaged-jobs/#slack) pre-packaged job.

<Step label="1">

### Create a Slack webhook URL

To send messages from EAS Workflows to our Slack channel, we need a webhook URL from Slack's settings:

1. Go to [api.slack.com/apps](https://api.slack.com/apps) and create a new Slack app (or use an existing one).
2. Under **Features**, select **Incoming Webhooks** and toggle it on.
3. Click **Add New Webhook** and select the channel where you want notifications.
4. Copy the webhook URL. It looks like: `https://hooks.slack.com/services/TD000/B000/XXXXXXXXXX`

</Step>

<Step label="2">

### Add a Slack notification job

Let's update **preview.yml** to add a `notify` job after the build jobs. The `notify` job uses a `webhook_url` and a `message` (or a `payload` for rich formatting using [Slack Block Kit](https://docs.slack.dev/block-kit)) parameters. We have to add the webhook URL directly in the workflow file.

```yaml preview.yml
name: Preview builds

jobs:
  fingerprint:
    # ...
  get_android_build:
    # ...
  get_ios_build:
    # ...
  build_android:
    # ...
  build_ios:
    # ...
  build_ios:
    # ...
  # @info #
  notify:
    name: Notify on Slack
    after: [build_android, build_ios]
    type: slack
    params:
      webhook_url: https://hooks.slack.com/services/TD000/B000/XXXXXXXXXX
      message: "Preview builds are ready - Android: ${{ after.build_android.status }}, iOS: ${{ after.build_ios.status }}"
  # @end #
```

In the above workflow, using `after` keyword, allows us to only trigger a notification when builds have completed either successfully, failed, or skipped.

The `message` uses `${{ after.build_android.status }}` and `${{ after.build_ios.status }}` to include the actual outcome (success, failure, or skipped) for each platform so the team knows exactly what happened within Slack.

</Step>

<Step label="3">

### Run the workflow

Run the workflow manually using the following command:

<Terminal cmd={['$ eas workflow:run .eas/workflows/preview.yml']} />

On EAS dashboard, you can verify that "Notify on Slack" job runs successfully.

## PR previews using EAS Update

When a PR reviewer looks at a pull request, they can read the diff but cannot see the changes visually or test them on a device without checking out the branch and starting a development server locally.

Traditionally, the problem described above is solved by deploy previews. When a pull request is created, a CI/CD workflow runs and generates the preview link for the reviewers. They can click and test the changes live. This is also know as preview links or staging environment.

On mobile for Android and iOS apps, the equivalent for web deployment previews can be set up using [EAS Update](/eas-update/introduction/). It is a service that delivers JavaScript bundle to devices that already have a compatible build installed. Publishing an update is fast because it skips the native compilation step. For PR reviews, we want an update and not a build because reviewers should be able to test and provide feedback to the PR.

To achieve this in EAS Workflows, we will use [`github-comment`](/eas/workflows/pre-packaged-jobs/#github-comment). It's a pre-packaged job that posts a comment on a pull request after the update publishes. This gives reviewers a link (or a QR code to scan) to open the update on their device.

<Step label="1">

### Create a pr-preview.yml

Inside **.eas/workflows/**, let's add a new file called **pr-preview.yml**. This workflow runs on creating a pull request using `on.pull_request` trigger. It only publishes an update using the `update` job type and then posts a comment on the PR with the link to the update.

```yaml pr-preview.yml
name: PR Preview

on:
  pull_request:
    branches: ['*']

jobs:
  publish_preview:
    name: Publish PR preview update
    type: update
    environment: preview
    params:
      channel: preview
  comment:
    needs: [publish_preview]
    type: github-comment
```

> **important** Before testing the above workflow, ensure that **pr-preview.yml** file is committed and pushed to the `main` branch of your GitHub repository. **EAS Workflows reads workflow files from the default branch.**

</Step>

<Step label="2">

### Test with a pull request

Let's test the workflow by creating a pull request in our GitHub repository. A pull request requires a separate branch from the `main`. Ru the following command to create and switch to a new branch:

<Terminal cmd={['$ git checkout -b test-preview']} />

Now, let's a visible change in the app (for example, change a text string or background color). Then, commit, and push the branch:

git add .
git commit -m "Test PR preview"
git push origin test-preview

  cmd={['$ git add .', '$ git commit -m "Test PR preview"', '$ git push origin test-preview']}
  ccmdCopy="git add . && git commit -m 'Test PR preview' && git push origin test-preview"

Open a pull request from `test-preview` to `main` on GitHub. The workflow will run automatically after we have created a pull request. On the EAS dashboard, we can verify that the `publish_preview` job runs first and the `comment` job runs after it succeeds.
