> **important** This library is currently in [alpha](/more/release-statuses/#alpha) and will frequently experience breaking changes.

`expo-app-intents` allows your app to expose Apple App Intents while keeping the intent declarations in Swift using [inline modules](/modules/inline-modules-reference/), where Apple's build-time metadata extraction can find them. The package provides the JavaScript API, native invocation queue, entity storage, config plugin, and a starter **app-intents** directory for native inline module code.

App Intent types cannot be created dynamically from JavaScript at runtime. You have to declare `AppIntent`, `AppEntity`, `EntityQuery`, and `AppShortcutsProvider` classes in Swift, then use this package to hand invocations to JavaScript and keep dynamic entity values in sync.

## Limitations

With `expo-app-intents`, you can implement many Apple Intelligence features. As of now, there is no way to support [on-screen intelligence](https://developer.apple.com/documentation/appintents/providing-contextual-cues-to-apple-intelligence-and-siri).

## How it works

App Intents are different from most Expo APIs because Apple discovers them at build time. The Swift types that describe your intents, entities, queries, and shortcut phrases must be compiled into the app target so Xcode's App Intents metadata processor can extract them. `expo-app-intents` uses Expo inline modules for that app-target Swift and provides a regular Expo package for the JavaScript API, native storage, and event delivery.

The generated **app-intents** directory is watched by Expo inline modules and contains your app-owned Swift:

- `AppShortcuts.swift` defines your concrete `AppShortcutsProvider`. Shortcut phrases are static metadata: they are compiled into the app and cannot be added dynamically from JavaScript.
- `AppIntentsSetup.swift` connects the app-target `AppShortcutsProvider` to the `expo-app-intents` runtime so JavaScript can request `AppShortcuts.updateAppShortcutParameters()` via the `expo-app-intents` package.
- The remaining Swift files define concrete `AppIntent`, `AppEntity`, and `EntityQuery` types. They compile into the app target, but are not registered as Expo modules unless they contain an Expo module definition.

When an intent runs, its `perform()` method should finish the native work that Siri or Shortcuts expects and then dispatch the intent to JavaScript/TypeScript, for example:

```swift
await AppIntentDispatcher.shared.dispatch(
  name: "orderFood",
  params: ["dishId": .string(dish.id), "dishName": .string(dish.name)]
)
```

`AppIntentDispatcher` is a Swift actor. It records the invocation in native storage first, then emits an `onIntent` event if the JavaScript runtime is already observing. This means intent delivery is at-least-once: JavaScript should handle each invocation by `id` and call `removePendingInvocationAsync(id)` after the action has been applied.

If Siri or Shortcuts runs while the app is cold, the event cannot be delivered immediately. The invocation stays in the pending queue. When your app starts, `useAppIntents()` calls your handler once with the current pending invocations and `newIntent` set to `null`; later live invocations call the handler again with the new invocation. You can also read the queue manually with `getPendingInvocationsAsync()`.

Parameterized intents use native entities. For example, the restaurant template defines a `DishEntity` and `DishQuery` in Swift, while JavaScript supplies the current dish list with `setEntityCatalogAsync('dish', dishes)`. The native query reads that catalog through `AppIntentEntityStore.shared`, so Siri and Shortcuts can resolve spoken values by ID, title, or synonym. Calling `setEntityCatalogAsync()` replaces the catalog and requests a shortcut parameter refresh; call it again whenever those values change.

Assistant schema intents, such as the journal example, are also defined in app-target Swift. The schema intent can be discovered directly on supported iOS versions, but the template includes a shortcut-facing wrapper intent so the main `AppShortcutsProvider` can remain available on devices below iOS 18. If your app only supports iOS 18 or later for that provider, you can reference the schema intent directly and remove the wrapper.

## Usage

After installing the package, run the initializer from your project root:

<Terminal cmd={['$ npx expo-app-intents init']} />

This command adds the config plugin, enables inline modules for the top-level **app-intents** directory, and writes starter Swift files. Then rebuild the native project:

<Terminal cmd={['$ npx expo prebuild -p ios', '$ npx expo run:ios']} />

The app intents are now set up in your app. Depending on your choice of examples, you can now add handlers or populate the app intents on the JavaScript side:

<Collapsible summary="Handle the counter example">

The counter example dispatches an `increaseCounter` invocation. Mount the hook once near the root of your app so it can process both pending invocations captured while JavaScript was cold and live invocations received while the app is running.

```tsx
import { useState } from 'react';
import { Text, View } from 'react-native';
import * as AppIntents from 'expo-app-intents';

export function CounterIntentHandler() {
  const [count, setCount] = useState(0);
  const [lastIntentId, setLastIntentId] = useState<string | null>(null);

  AppIntents.useAppIntents(async pendingIntents => {
    for (const invocation of pendingIntents) {
      if (invocation.name !== 'increaseCounter') {
        continue;
      }

      setCount(value => value + 1);
      setLastIntentId(invocation.id);
      await AppIntents.removePendingInvocationAsync(invocation.id);
    }
  });

  return (
    <View>
      <Text>Counter: {count}</Text>
      {lastIntentId ? <Text>Last opened by Siri: {lastIntentId}</Text> : null}
    </View>
  );
}
```

</Collapsible>

<Collapsible summary="Handle the restaurant example">

The restaurant example uses a dynamic `dish` entity catalog. Seed the catalog from JavaScript after app startup, then handle `orderFood` invocations. Calling `setEntityCatalogAsync()` replaces the native catalog and refreshes shortcut parameters so Siri and Shortcuts can resolve dishes by ID, title, or synonym.

```tsx
import { useEffect, useState } from 'react';
import { Text, View } from 'react-native';
import * as AppIntents from 'expo-app-intents';

const dishes = [
  { id: 'margherita-pizza', title: 'Margherita Pizza', synonyms: ['margherita', 'pizza'] },
  { id: 'spaghetti-carbonara', title: 'Spaghetti Carbonara', synonyms: ['carbonara'] },
  { id: 'tiramisu', title: 'Tiramisu', synonyms: ['dessert'] },
];

export function RestaurantIntentHandler() {
  const [latestOrder, setLatestOrder] = useState<string | null>(null);

  useEffect(() => {
    void AppIntents.setEntityCatalogAsync('dish', dishes);
  }, []);

  AppIntents.useAppIntents(async pendingIntents => {
    for (const invocation of pendingIntents) {
      if (invocation.name !== 'orderFood') {
        continue;
      }

      setLatestOrder(String(invocation.params.dishName ?? invocation.params.dishId));
      await AppIntents.removePendingInvocationAsync(invocation.id);
    }
  });

  return (
    <View>
      <Text>{latestOrder ? `Latest order: ${latestOrder}` : 'No orders yet.'}</Text>
    </View>
  );
}
```

</Collapsible>

<Collapsible summary="Handle the journal example">

The journal example dispatches `createJournalEntry`. The shortcut-facing intent sends `id`, `title`, and `message`; the schema-domain intent can also include fields such as `entryDate`, `location`, and `mediaItemCount`.

```tsx
import { useState } from 'react';
import { FlatList, Text } from 'react-native';
import * as AppIntents from 'expo-app-intents';

type JournalEntry = {
  id: string;
  title: string;
  message: string;
};

export function JournalIntentHandler() {
  const [entries, setEntries] = useState<JournalEntry[]>([]);

  AppIntents.useAppIntents(async pendingIntents => {
    for (const invocation of pendingIntents) {
      if (invocation.name !== 'createJournalEntry') {
        continue;
      }

      const entry = {
        id: String(invocation.params.id ?? invocation.id),
        title: String(invocation.params.title ?? 'Untitled'),
        message: String(invocation.params.message ?? ''),
      };

      setEntries(current => [entry, ...current]);
      await AppIntents.removePendingInvocationAsync(invocation.id);
    }
  });

  return (
    <FlatList
      data={entries}
      keyExtractor={entry => entry.id}
      renderItem={({ item }) => (
        <Text>
          {item.title}: {item.message}
        </Text>
      )}
    />
  );
}
```

</Collapsible>

```tsx Basic example
import * as AppIntents from 'expo-app-intents';

export function AppIntentHandler() {
  AppIntents.useAppIntents(async (pendingIntents, newIntent) => {
    for (const invocation of pendingIntents) {
      switch (invocation.name) {
        case 'increaseCounter':
          break;
        case 'orderFood':
          console.log('Dish:', invocation.params.dishName, invocation.id === newIntent?.id);
          break;
      }

      await AppIntents.removePendingInvocationAsync(invocation.id);
    }
  });

  return null;
}
```

Feed dynamic values for parameterized intents from JavaScript/TypeScript:

```ts
await AppIntents.setEntityCatalogAsync('dish', [
  { id: 'margherita-pizza', title: 'Margherita Pizza', synonyms: ['margherita', 'pizza'] },
  { id: 'spaghetti-carbonara', title: 'Spaghetti Carbonara', synonyms: ['carbonara'] },
]);
```

The generated entity queries read those values through `await AppIntentEntityStore.shared.entities(ofKind:)`, so Siri and Shortcuts can offer and resolve them as intent parameters.

Classic App Shortcut phrases can interpolate at most one non-array parameter. The generated restaurant order example includes a parameterized phrase such as "Place an order for Tiramisu in MyApp" for one-shot Siri commands, plus static phrases such as "Place an order in MyApp" and "Choose Italian food in MyApp" so Siri can ask which dish you want. Parameterized phrases can create pre-filled tiles in Shortcuts; if a tile was created by an older build and runs stale parameter values, delete that tile so iOS can recreate it from the current app metadata.

## Platform constraints

- Requires a native iOS build. `expo-app-intents` is not available in Expo Go.
- App Shortcut phrases are compiled into the app and cannot be added dynamically from JavaScript.
- Every App Shortcut phrase must include `\(.applicationName)`.
- A single App Shortcut phrase can interpolate at most one non-array parameter.
- Apps can define at most 10 App Shortcuts.
- Assistant schema examples require iOS 18 or later and must match one of Apple's [supported App Intent domains](https://developer.apple.com/documentation/appintents/app-schema-domains).

## Additional resources

To learn how to use `expo-app-intents`, start with the basic examples that `npx expo-app-intents init` scaffolds. The counter example is the simplest. Once you understand it, you understand the foundation `expo-app-intents` is built on. From there, move on to the restaurant and journal examples for more practical usage.

The main goal of the examples is to demonstrate the process of connecting the JS and Swift runtimes. Once you master those examples, you should consider learning more about App Intent development from Apple's resources:

- [Accelerating app interactions with App Intents](https://developer.apple.com/documentation/appintents/acceleratingappinteractionswithappintents)
- [Code-along: Make your app available to Siri](https://developer.apple.com/videos/play/wwdc2026/344/)
